### Task 4 — Multi-Agent System (25 pts)

Build a multi-agent system using `run_specialist_agent()` as the building block for each agent. The architecture is your decision. Three patterns to consider:

**Orchestrator + Specialists + Critic**
An orchestrator reads the question and delegates sub-tasks to domain specialists. A critic fact-checks each specialist's answer against its raw tool data. A synthesizer merges everything into a final answer. Most transparent, most API calls, highest latency.

**Sequential Pipeline**
Agents chain in order — each receives the previous agent's output as context. Simpler to implement, but errors in early steps propagate.

**Parallel Specialists + Aggregator**
All specialists run on the question simultaneously (use `concurrent.futures.ThreadPoolExecutor`). An aggregator merges the results. Fastest, but no cross-checking between agents.

You can combine elements from multiple patterns. Whatever you build, `run_multi_agent()` must return:

```python
{
    "final_answer"  : str,          # the answer shown to the user
    "agent_results" : list,         # list[AgentResult] — one per specialist
    "elapsed_sec"   : float,        # total wall clock time
    "architecture"  : str,          # short label e.g. "orchestrator-critic"
}
```

Document your design decisions in comments — the grading includes your reasoning.

**Suggested tool subsets per specialist** (adjust to fit your design):

### Task 5 — Evaluator (15 pts)

An LLM-as-judge that scores each agent answer 0–3. This runs automatically inside the evaluation runner and fills the score, reasoning, hallucination, and issues columns in the Excel file.

**Required return format:**
```python
{
    "score"                 : int,       # 0, 1, 2, or 3
    "max_score"             : int,       # always 3
    "reasoning"             : str,       # one sentence explanation
    "hallucination_detected": bool,      # True if answer contains invented facts
    "key_issues"            : list[str], # specific problems; empty list if none
}
```

**Scoring rubric to include in your prompt:**
```
3 — Fully correct:     all required data present and accurate, conditions met
2 — Partially correct: key data present but incomplete or minor errors
1 — Mostly wrong:      wrong numbers, missed conditions, or suspected fabrication
0 — Complete failure:  refused, gave up, or answered an unrelated question
```

**Hallucination flags — what to instruct the model to catch:**
- Specific prices, P/E ratios, or % changes cited with no tool call to support them
- Tickers that don't exist or aren't relevant to the question
- Definitive claims about "current" data from an agent that called no live data tools

**Important notes:**
- The evaluator only sees text — it cannot verify numbers against a live data source. Its hallucination detection is probabilistic, not ground truth. You will analyse its limitations in the reflections.
- Strip markdown fences before parsing the JSON response.
- Return the fallback dict on any parse error — do not let the evaluation runner crash.
- The three calibration tests in the notebook must all pass before the full run.

## Known Issues

**Sector column values** — the DB stores `"Information Technology"` not `"Technology"`, and `"Financials"` not `"Finance"`. Check the printed output of `create_local_database()` to see exact values before implementing `get_tickers_by_sector`.

**Alpha Vantage rate limiting** — the free tier allows ~5 requests/minute. If P/E ratio calls return empty responses, add `time.sleep(1)` between ticker requests inside `get_company_overview`.

**Delisted tickers** — a small number of S&P 500 tickers have since been acquired or delisted. `yfinance` raises exceptions or returns empty data for these. Wrap each ticker's download in `try/except` and return `{"error": "possibly delisted"}`.

**yfinance FutureWarning** — use `.iloc[0].item()` instead of `.iloc[0]` when extracting scalar values from a DataFrame column to suppress deprecation warnings on sector-wide queries.
